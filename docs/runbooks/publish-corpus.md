# Publish Corpus

This runbook covers the `M0-M1` dataset control plane. It publishes immutable corpus artifacts and updates the preview or production dataset pointer. It does not change the live browse runtime yet; runtime consumption moves in `M3`.

## Support Matrix

- Bun: `>=1.3.6`
- Node: `>=25.0.0`
- Wrangler package: `^4.72.0`
- Local control-plane commands: `bun run validate-dataset`, `bun run publish-dataset`, `bun run promote-dataset`, `bun run rollback-dataset`, `bun run prune-datasets`

## Required Inputs

- `HF_TOKEN`
- `HF_EXCERPT_STORE`, `HF_ASL_STORE`, `HF_SHAMELA4_STORE`
- `HF_EXCERPT_REVISION`, `HF_ASL_REVISION`, `HF_SHAMELA4_REVISION` for release builds
- `ILMTEST_API_URL`
- `R2_BUCKET`
- `R2_ENDPOINT` or `R2_ACCOUNT_ID` / `CF_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`

## Local Flow

1. Generate or refresh local corpus data:
   `bun run setup <collection ids...>`
2. Validate the local dataset inputs:
   `bun run validate-dataset`
3. Publish the immutable dataset:
   `bun run publish-dataset -- --dataset-version <optional-version>`
4. Validate the remote dataset by version if you want an explicit post-publish check:
   `bun scripts/validateDataset.ts remote --dataset-version <dataset-version>`
5. Promote the channel pointer:
   `bun run promote-dataset -- --channel preview --dataset-version <dataset-version>`
   `bun run promote-dataset -- --channel prod --dataset-version <dataset-version>`
6. Prune old preview datasets after a successful preview publish:
   `bun run prune-datasets`

## Notes

- `tmp/dataset-build/metadata.json` is generated by `setup.ts` and is the publisher input contract.
- Resume state is stored in `tmp/publish-state/<datasetVersion>.json`.
- The legacy Pages deployment flow still exists for continuity, but it is not the canonical corpus publish path anymore.
- Emergency fallback is `rclone sync`, but it is non-canonical because it does not provide manifest and promotion semantics.
